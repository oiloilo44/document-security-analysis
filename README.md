## 1. 프로젝트 개요
본 프로젝트는 기업용 문서 보안 시스템(DRM)의 동작 방식을 심층 분석하고, 그 구조적 한계를 파악하여 우회할 수 있는 취약점을 식별한 보안 연구 포트폴리오입니다.

초기에는 웹 서버 통신을 통한 복호화 API 모방을 시도하였으나, 서버단 통합 암호화 로직 한계로 인해 클라이언트 사이드 리버싱 및 **'Trusted Process Hollowing (신뢰된 프로세스 할로잉)'** 기법을 통한 메모리 후킹으로 전략을 선회하여 최종적인 개념 증명(PoC)에 성공한 사례입니다.

## 2. 보안 시스템(DRM) 동작 메커니즘 분석 및 한계점 극복 과정

### Phase 1: 서버사이드 인증 및 Web API 분석 (우회 불가 확인)
- **접근 방식:** 네트워크 프록시 분석을 통해 권한 승인 및 파일 업로드 API(`POST /gateway/.../upload`)의 프로토콜 분석.
- **분석 결과:** 파일 업로드 시 필요한 인증 토큰(JWT)과 세션 키는 서버와의 복잡한 상호 작용을 통해 생성되며, 클라이언트 사이드 모방만으로는 한계가 명확함.
- **결정:** 서버를 속이는 방식은 배제하고, 로컬 환경에서 발생하는 파일 I/O를 직접 모니터링 및 탈취하기로 방향 전환.

### Phase 2: 로컬 DRM 모듈(DLL) 정적/동적 분석 시도
- **접근 방식:** 윈도우 OS에 설치된 로컬 암/복호화 엔진(예: `[REDACTED]_Packager.dll` 등)의 코어 API(`DoExtract` 계열)를 직접 호출하는 방식 테스트.
- **분석 결과:** DLL 자체의 암호화는 풀려있으나, 내부적으로 라이선스 체크 및 윈도우 커널 단말기 UUID 등을 포함한 `initialize_all` 권한 제약이 걸려 있음. C ctypes로 랩핑하여 호출 시 Silent Reject 반환(`[ERROR_CODE]`).
- **결정:** 권한 검증 분기를 단순 JMP 변조(Patch)하는 것은 런타임 크래시를 유발하므로, **DRM 드라이버가 이미 신뢰하는(Trusted) 프로세스 내부로 침투하는 방법**으로 최종 피벗(Pivot).

### Phase 3: 최종 공격 벡터 - Trusted Process 에 인젝션 (On-the-fly Decryption 역이용)
- **개념 파악:** 파일 시스템 드라이버(Minifilter) 단에서 동작하는 DRM은 오직 허가된 뷰어(Excel, Word 등)가 파일 입출력을 발생시킬 때만 투명하게 **스트림 복호화(On-the-fly Decryption)**를 제공함.
- **공격 시나리오:** 
  1. Python 기반의 런처 설계 및 대상 뷰어를 백그라운드 프로세스로 실행(또는 기존 프로세스에 Attach).
  2. 동적 인스트루멘테이션 프레임워크(Frida)를 활용하여 뷰어 프로세스 내부 메모리에 JS 페이로드 인젝션.
  3. 뷰어 영역 내에서 `kernel32.dll`의 `ReadFile` 등 Native WinAPI 호출.
  4. 커널 필터는 이를 정상 뷰어의 접근으로 간주하고, 버퍼에 평문 데이터 반환.
  5. 평문 데이터를 즉시 가로채 별도 버퍼를 통해 덤프.

## 3. PoC 도구 아키텍처 및 구현 (Python + Frida)

단순한 개념 증명을 넘어, 대규모 환경에서도 안정적으로 동작할 수 있도록 **다중 프로세스 병렬 인젝션 프레임워크**를 구현하였습니다.

### 주요 기술 스택
- **Language:** Python 3.x, JavaScript
- **Framework:** Frida (Dynamic Binary Instrumentation)
- **Concurrency:** `concurrent.futures.ThreadPoolExecutor`, `threading.Event`

### 핵심 구현 로직 (Architectural Highlights)

1. **병렬 어태치 및 멀티 프로세스 동시 제어 (`-a` mode)**
   - `psutil`을 통해 현재 구동 중인 오피스 애플리케이션들을 자동 탐지.
   - 열린 파일 핸들(`open_files()`) 중 복호화 대상 확장자를 선별.
   - 워커 스레드풀을 생성하여 탐지된 다수의 뷰어 프로세스에 동시 인젝션 수행 (병목 방지).

2. **재암호화 패킹 정책 회피(Bypass Re-encryption)**
   - DRM 드라이버가 덤프된 평문 파일을 다시 암호화하는 것을 막기 위해, 출력 파일 저장 시 I/O 모니터링 조건에 걸리지 않도록 고의적으로 위장 확장자(`.dat`)를 사용.

3. **호환성 및 폴백(Fallback) 라우팅**
   - 32bit/64bit 프로세스 아키텍처 불일치 예외 처리 완료.
   - 다양한 버전의 Frida/OS 환경에서 발생할 수 있는 내부 API 참조 에러(`Module.findExportByName` undef)를 방어하기 위한 3단계 모듈 트리 탐색 로직 적용.

## 4. 시사점 및 레슨 런 (Lessons Learned)
* **보호 기법의 그림자:** 시스템이 제공하는 '투명한 호환성(앱 수정 없이 자동 복호화)'은 양날의 검으로 작용하며, 사용자 편의를 위한 화이트리스트(Trusted App) 구조가 결과적으로 가장 치명적인 공격 지점이 될 수 있음을 확인.
* **유연한 전환(Agility)의 중요성:** 정적 분석(리버싱)에서 동적 분석(메모리 덤프), 그리고 최종적으로 운영체제의 아키텍처를 역이용하는 방식까지, 단일 돌파구에 매몰되지 않고 넓은 시야로 공격 기법을 융합하는 것이 중요함.

> **Note:** 본 레파지토리의 파이썬 스크립트(`drm_hollowing_poc.py`)는 핵심 해킹 로직이 마스킹 처리된 프레임워크 구조 지표 코드입니다.
